Fixed: `Box[T]`'s bare-type-parameter field, when `T` resolves to a relation (`Box[@@Employee]`), now reloads correctly instead of failing at runtime. `_plain_struct_from_json_call` detects (via `_type_involves_relation`, substituting `T` into `Box`'s own raw fields before checking) whenever a generic plain struct's instantiation reaches a relation, and routes to a distinct, fully-monomorphized `sqrrl__<Name><_MonoSuffix>_from_json` companion instead of the ordinary generic one -- generated (`_collect_mono_plain_struct_targets`/`_substituted_fields_for`) from the struct's own *substituted* field list, so a relation hiding behind `T` is dispatched exactly like a hand-written relation field, with the right sibling table threaded through both the call site and the companion's own signature. The ordinary generic companion (`sqrrl__Box_from_json[T]`) is still emitted unchanged, for any relation-free instantiation. Verified via `examples/plain_structs` (`@@Person`'s new `@@box: Box[@@Employee]` field) and a unit test (`test_emit_json_module_generic_plain_struct_relation_field_gets_monomorphized_companion`).

Multi-type-parameter generics: confirmed via a scratch spike (`Pair[K, V]` with `key: K, value: V`, instantiated as `Pair[@@Employee, String]`) that the mechanism generalizes correctly -- only the parameter that actually resolves to a relation gets relation-handle dispatch in the monomorphized companion; any other parameter (however many) still gets ordinary leaf/plain-struct treatment in that same function. Not added as a permanent example (Box already covers the mechanism end-to-end); worth promoting if a real multi-param generic plain struct shows up in practice.

Separately (pre-existing, unrelated to the above): `is_wrapped_relation_type`'s own `@@`-marking-detection check only ever looks at a parameterized type's *first* type argument (`parser/relation_type_text.mojo`) -- a field whose relation sits in a later argument (`Pair[String, @@Employee]`) gets rejected at parse time with "@@ marking must match between field name and type", even though the field is genuinely relation-shaped once you look past the first argument. Only `Dict`-style "the interesting argument is always the first one" conventions are covered today. Worth a follow-up if a real field ever needs a relation in a non-first argument position.
