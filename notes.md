New nuance, worth flagging: Box[T]'s bare-type-parameter field, when T specifically resolves to a relation (Box[@@Employee]), dumps fine but fails reload — and now at runtime, not DSL-compile-time. I verified this directly: dump produces {"box":{"value":0}} correctly (reflection handles it), but reload raises sqrrl__from_json: unsupported type at runtime. The cause: the sibling-table-discovery pass that decides which tables Box's own generated from_json needs never substitutes type parameters, so it never realizes Box[@@Employee] needs Employee's table — and even if it did, the shared dispatcher has no way to accept one. Before this session's rearchitecture, every bare-type-param field raised a clear compile-time error regardless of what T was; now only this one relation-specific case still fails, and it fails later and less clearly than before. Worth a follow-up if you want a clean compile-time error restored for just this case.

Fixing this properly would mean monomorphizing Box's own from_json per distinct instantiation actually used (generating a separate, concrete companion per Box[@@SomeTarget], the way you'd expect a template to specialize) rather than the one generic function it gets today — a real, scoped piece of additional work, not a fundamental dead end like the reflection question was.
